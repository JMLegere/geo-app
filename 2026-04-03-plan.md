# EarthNova v3 Rebuild Plan

> 2026-04-03. Nuke all Dart code, write master design doc, rebuild from scratch with TDD and observability from day 1.

---

## Context

EarthNova has been broken for weeks — users can't use the app, root cause unknown. 16 commits of features got stacked on the v2 rewrite (commit `3569cc8`) and complexity returned. The codebase is 23,400 lines across ~130 Dart files with 20 providers, 8 screens, and a 6-table Drift schema. Rather than debug the stack, we nuke it and rebuild the MVP: login + pack.

### Decisions (locked)

| Decision | Detail |
|----------|--------|
| MVP scope | Login screen + Pack screen showing existing species data from Supabase. Everything else stubbed. |
| Auth | Phone number → derived email+password via Supabase. No OTP. |
| Data | Don't reset beta accounts. New v3 Supabase tables alongside old ones. Migrate data. Keep old tables as safety net. |
| PR #412 | Abandon (108-test PR on `test/comprehensive-coverage`). |
| Stack | Same — Flutter 3.41.3, Riverpod 3.2.1 (Notifier), Supabase. No Drift, no SQLite, no map libs. |
| Observability | Every state transition logs to Supabase `app_logs` from commit 1. |
| Everything as code | Design doc, ADRs, runbook, post-mortems, error catalog, schema snapshots, seed data, env inventory, dependency justification, performance budgets — all in the repo. |
| No feature flags | Only prod. |

### Issue Resolutions

7 issues were identified in the oracle audit. All resolved:

**Issue 1 (Critical): `_deriveEmail`/`_derivePassword` verified against actual v2 code.**

Confirmed at `lib/data/sync/supabase_auth.dart:145-153`:
```dart
static String _deriveEmail(String phone) {
  final digits = phone.replaceAll(RegExp(r'[^\d]'), '');
  return '$digits@earthnova.app';
}

static String _derivePassword(String phone) {
  final bytes = utf8.encode('$phone:earthnova-beta-2026');
  return sha256.convert(bytes).toString();
}
```
These MUST be byte-for-byte identical in v3. A regression test with known phone→email→password values is mandatory.

**Issue 2 (High): Migration SQL column names verified against actual schema.**

`item_instances` actual columns (from migrations 002, 006, 023):
- Core: `id`, `user_id`, `definition_id`, `affixes`, `acquired_at`, `acquired_in_cell_id`, `daily_seed`, `status`, `created_at`, `updated_at`
- Denormalized (006): `display_name`, `scientific_name`, `category_name`, `rarity_name`, `habitats_json`, `continents_json`, `taxonomic_class`
- Denormalized (023): `icon_url`, `art_url`, `animal_class_name`, `food_preference_name`, `climate_name`, `size_name`, `brawn`, `wit`, `speed`, plus location fields and enrichver stamps

The oracle's migration SQL used the correct column names. Confirmed: `ii.display_name`, `ii.scientific_name`, `ii.category_name`, `ii.rarity_name`, `ii.icon_url`, `ii.art_url`, `ii.status` all exist.

**Issue 3 (Medium): `railway.toml` — Railway doesn't use config files.**

Railway auto-detects Dockerfile. No `railway.toml` format exists. Deploy config is: push to main → Railway watches repo → builds Dockerfile → deploys. This is already documented in the Dockerfile itself. Added to runbook instead of creating a separate file.

**Issue 4 (Medium): Anonymous sessions — already handled in v2.**

`supabase_auth.dart:186-191` already detects anonymous sessions and signs them out:
```dart
final isAnonymous = user.userMetadata?['is_anonymous'] == true ||
    (user.appMetadata['provider'] == 'anonymous');
if (isAnonymous) {
  await _auth.signOut();
  return false;
}
```
v3 keeps this behavior. Anonymous users are signed out and shown the login screen. No data migration needed for anonymous sessions (they have no meaningful data).

**Issue 5 (Medium): Edge Functions reference old tables.**

`validate-encounter` and `process-enrichment-queue` write to `item_instances`, not `v3_items`. These functions are deferred — they support discovery and enrichment, which are not in the MVP. When discovery is rebuilt (post-MVP), the functions will be updated to write to v3 tables. For now, they remain untouched and non-breaking (they just won't affect v3 data).

**Issue 6 (Low): `StateProvider` for auth service injection.**

Changed to `Provider<AuthService>` with override pattern:
```dart
final authServiceProvider = Provider<AuthService>((ref) => MockAuthService());
// In main.dart: authServiceProvider.overrideWithValue(supabaseAuthService)
// In tests: authServiceProvider.overrideWithValue(mockAuthService)
```

**Issue 7 (Low): IndexedStack for TabShell.**

Acceptable for MVP. Only 4 tabs — 2 are trivial stubs ("Coming soon" text), 1 is settings (one button), 1 is pack. Total widget cost is negligible. The GBA Rule applies when there are expensive widgets (map, fog overlay, animations). None of those exist in v3 MVP. Noted in design doc.

**Sections:**

1. **Product Vision** — One paragraph: what EarthNova is, who it's for, v3 goal.

2. **MVP Scope** — What ships (login, pack, stubs, observability). What doesn't ship (map, fog, GPS, discovery, sanctuary, offline, SQLite, enrichment, achievements, streaks).

3. **Architecture** — Layer diagram: `screens/ → providers/ → services/ → Supabase`. No domain layer, no engine, no repositories. Target file structure (~22 files in lib/).

5. **Data Model** — 4 Supabase tables (v3_profiles, v3_items, v3_cell_visits, v3_write_queue). Dart models: AuthState, UserProfile, Item. No local SQLite.

6. **Observability** — Reuse existing `app_logs` table. Event catalog (13 events across auth, data, lifecycle, error categories). ObservabilityService: batched writes, 5s flush, fire-and-forget. Session ID per app launch.

7. **Error Handling** — Error catalog reference. Every error has a code, user message, cause, and resolution.

8. **Performance Budgets** — Login < 2s, pack load < 500ms, bundle < 3MB gzipped.

9. **Acceptance Criteria** — Per-feature checklists that map 1:1 to test cases. Auth (7 criteria), Pack (7 criteria), Stubs (3 criteria), Infrastructure (6 criteria).

### Step 2.1 — Write migration

**File:** `supabase/migrations/032_v3_schema.sql`

```sql
-- ============================================================================
-- Migration 032: v3 schema — clean tables for the EarthNova v3 rebuild.
-- Old tables remain untouched.
-- ============================================================================

-- v3_profiles
CREATE TABLE v3_profiles (
  id            UUID REFERENCES auth.users PRIMARY KEY,
  phone         TEXT NOT NULL DEFAULT '',
  display_name  TEXT NOT NULL DEFAULT 'Explorer',
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE v3_profiles ENABLE ROW LEVEL SECURITY;
CREATE POLICY "v3_profiles_select_own" ON v3_profiles FOR SELECT USING (auth.uid() = id);
CREATE POLICY "v3_profiles_insert_own" ON v3_profiles FOR INSERT WITH CHECK (auth.uid() = id);
CREATE POLICY "v3_profiles_update_own" ON v3_profiles FOR UPDATE USING (auth.uid() = id);

-- v3_items
CREATE TABLE v3_items (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id             UUID REFERENCES auth.users NOT NULL,
  definition_id       TEXT NOT NULL,
  display_name        TEXT NOT NULL,
  scientific_name     TEXT,
  category            TEXT NOT NULL DEFAULT 'fauna',
  rarity              TEXT,
  icon_url            TEXT,
  icon_url_frame2     TEXT,
  art_url             TEXT,
  acquired_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  acquired_in_cell_id TEXT,
  status              TEXT NOT NULL DEFAULT 'active',
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE v3_items ENABLE ROW LEVEL SECURITY;
CREATE POLICY "v3_items_select_own" ON v3_items FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "v3_items_insert_own" ON v3_items FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "v3_items_update_own" ON v3_items FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "v3_items_delete_own" ON v3_items FOR DELETE USING (auth.uid() = user_id);

CREATE INDEX idx_v3_items_user_id ON v3_items(user_id);
CREATE INDEX idx_v3_items_definition_id ON v3_items(definition_id);
CREATE INDEX idx_v3_items_user_acquired ON v3_items(user_id, acquired_at DESC);

-- v3_cell_visits
CREATE TABLE v3_cell_visits (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES auth.users NOT NULL,
  cell_id     TEXT NOT NULL,
  visited_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(user_id, cell_id)
);

ALTER TABLE v3_cell_visits ENABLE ROW LEVEL SECURITY;
CREATE POLICY "v3_cell_visits_select_own" ON v3_cell_visits FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "v3_cell_visits_insert_own" ON v3_cell_visits FOR INSERT WITH CHECK (auth.uid() = user_id);

CREATE INDEX idx_v3_cell_visits_user_id ON v3_cell_visits(user_id);

-- v3_write_queue
CREATE TABLE v3_write_queue (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES auth.users NOT NULL,
  action      TEXT NOT NULL,
  payload     JSONB NOT NULL DEFAULT '{}'::jsonb,
  status      TEXT NOT NULL DEFAULT 'pending',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  resolved_at TIMESTAMPTZ
);

ALTER TABLE v3_write_queue ENABLE ROW LEVEL SECURITY;
CREATE POLICY "v3_write_queue_select_own" ON v3_write_queue FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "v3_write_queue_insert_own" ON v3_write_queue FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "v3_write_queue_update_own" ON v3_write_queue FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "v3_write_queue_delete_own" ON v3_write_queue FOR DELETE USING (auth.uid() = user_id);

-- Data migration: profiles → v3_profiles
INSERT INTO v3_profiles (id, phone, display_name, created_at, updated_at)
SELECT
  p.id,
  COALESCE(u.raw_user_meta_data->>'phone_number', u.phone, u.email, ''),
  COALESCE(p.display_name, 'Explorer'),
  p.created_at,
  p.updated_at
FROM profiles p
JOIN auth.users u ON u.id = p.id
ON CONFLICT (id) DO NOTHING;

-- Data migration: item_instances → v3_items
-- Column names verified against migrations 002 + 006 + 023
INSERT INTO v3_items (
  id, user_id, definition_id, display_name, scientific_name,
  category, rarity, icon_url, icon_url_frame2, art_url,
  acquired_at, acquired_in_cell_id, status, created_at
)
SELECT
  ii.id,
  ii.user_id,
  ii.definition_id,
  COALESCE(ii.display_name, ii.definition_id),
  ii.scientific_name,
  COALESCE(ii.category_name, 'fauna'),
  ii.rarity_name,
  COALESCE(ii.icon_url, s.icon_url),
  NULL,  -- icon_url_frame2: not yet generated, enrichment pipeline will populate
  COALESCE(ii.art_url, s.art_url),
  ii.acquired_at,
  ii.acquired_in_cell_id,
  COALESCE(ii.status, 'active'),
  ii.created_at
FROM item_instances ii
LEFT JOIN species s ON s.definition_id = ii.definition_id
ON CONFLICT (id) DO NOTHING;

-- Data migration: cell_progress → v3_cell_visits
INSERT INTO v3_cell_visits (user_id, cell_id, visited_at)
SELECT user_id, cell_id, COALESCE(last_visited, created_at)
FROM cell_progress
WHERE visit_count > 0
ON CONFLICT (user_id, cell_id) DO NOTHING;

-- Trigger: auto-create v3_profiles on new user signup
CREATE OR REPLACE FUNCTION public.handle_new_user_v3()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.v3_profiles (id, phone, display_name)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'phone_number', NEW.phone, NEW.email, ''),
    COALESCE(NEW.raw_user_meta_data->>'display_name', 'Explorer')
  )
  ON CONFLICT (id) DO NOTHING;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Both old and new triggers fire — user gets row in both profiles AND v3_profiles
CREATE TRIGGER on_auth_user_created_v3
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user_v3();
```
