# iOS Safari + MapLibre GL JS — Problem Space

> Full diagnostic record of the blank lavender screen bug on iOS Safari / Arc browser.
> Lives here so future sessions don't re-litigate settled decisions.

---

## The Symptom

iOS Safari (including Arc / WebKit) shows a **blank lavender screen** for the map.
Desktop Chrome: works. Android Chrome: works. iOS Safari: lavender.

The lavender is MapLibre's background/ocean colour — the map widget is alive but
no tiles ever render.

---

## Root Cause: Three Interlocking Constraints

All three must be satisfied simultaneously. Violating any one breaks the map.

### Constraint 1 — WebGL Context Order (iOS Safari limit)

iOS Safari allocates **~8–16 WebGL contexts per tab**. CanvasKit (Flutter's
renderer) and MapLibre GL JS both need WebGL contexts at startup.

**Rule:** CanvasKit must acquire its context(s) **before** MapLibre requests its
context. If MapLibre loads first (e.g. via a `<script>` in `<head>`), it may
exhaust the pool before CanvasKit can initialise. CanvasKit then silently fails
and the app never boots.

**Observable signal:**
- `health: canvas element missing` in `telemetry_logs`
- No `app.cold_start` Dart event

### Constraint 2 — maplibregl Must Exist Before First Flutter Frame

The `maplibre_gl` Flutter plugin (web) calls `new maplibregl.Map(...)` when its
`HtmlElementView` is first inserted into the DOM. This happens **during the first
Flutter frame** (inside `runApp()`).

If `window.maplibregl` is `undefined` at that moment:
- The plugin silently fails to create the map object
- The Flutter widget tree continues rendering without a map
- No error is surfaced — just a lavender container forever
- Even when MapLibre loads later, the dead widget doesn't recover

**Observable signal:**
- `flutter_bootstrap_complete` fires ✅
- `maplibre_lazy_loaded` fires ✅
- `ios_fix: WEBGL_provoking_vertex applied` fires ✅ (MapLibre later creates a *new* map for the iOS fix)
- But the screen stays lavender — the plugin's map object was never created

### Constraint 3 — The Inline Script Timing Problem

The standard approach to hooking Flutter's bootstrap is to monkey-patch
`window._flutter.loader.load`. But this **always silently fails** because:

1. Our inline `<script>` in `index.html` runs **synchronously** during HTML parse
2. `flutter_bootstrap.js` is loaded **async** — it hasn't run yet
3. At the time our script runs, `window._flutter` is `undefined`
4. `var origLoader = window._flutter && window._flutter.loader` → `undefined`
5. The `if (origLoader && origLoader.load)` guard silently exits — no patch

**Observable signal:**
- `maplibre_fallback_inject: timeout fallback` fires on every load (5s timeout
  is the only trigger — the callback path never ran)
- `flutter_bootstrap_complete` never appears

## What the Flutter Bootstrap Actually Does

```
flutter_bootstrap.js (generated by flutter build web):

  window._flutter || (window._flutter = {})
  window._flutter.loader || (window._flutter.loader = new FlutterLoader)

  // ← our patch must already be in place at this point

  _flutter.loader.load({
    serviceWorkerSettings: { serviceWorkerVersion: "..." }
  })
  // ↑ no onEntrypointLoaded — uses internal default:
  //   l => l.initializeEngine(config).then(u => u.runApp())
```

`flutter_bootstrap.js` both **creates** the loader and **calls** `load()`.
There is no external call in `index.html` to intercept.

## Failed Approaches (History)

### Attempt 1 — Synchronous `<script>` in `<head>` (v2 / original)

```html
<script src="maplibre-gl.js"></script>  <!-- line 35, before Flutter -->
```

**Problem:** Violates Constraint 1. MapLibre runs before CanvasKit, exhausts
WebGL contexts, CanvasKit never inits.

**Signal:** `health: canvas element missing`, no `app.cold_start`.

### Attempt 2 — Inject after `runApp()` + 100ms buffer (PR #454 / first attempt)

```javascript
await appRunner.runApp();
await new Promise(r => setTimeout(r, 100));
_injectMapLibre();
```

**Problem:** Violates Constraint 2. `MapLibreMap` widget builds *during*
`runApp()`. By the time MapLibre loads 100ms later, the plugin has already
silently failed. The lavender screen stays forever.

**Signal:** `flutter_bootstrap_complete` fires, `ios_fix` fires, but lavender.

### Attempt 3 — Direct monkey-patch with if-guard (PR #454 / second attempt)

```javascript
var origLoader = window._flutter && window._flutter.loader;
if (origLoader && origLoader.load) { ... }
```

**Problem:** Violates Constraint 3. `window._flutter` doesn't exist when this
runs. The if-guard silently exits every time.

**Signal:** `maplibre_fallback_inject: timeout fallback` on every load. No
`flutter_bootstrap_complete` event.

## The Correct Solution (PR #456 + follow-up)

**Inject MapLibre between `initializeEngine()` and `runApp()`, using an
`Object.defineProperty` setter to intercept the loader assignment.**

### Why `Object.defineProperty`

```javascript
window._flutter = window._flutter || {};
Object.defineProperty(window._flutter, 'loader', {
  set: function(loaderInstance) {
    // Wrap load() here — called immediately after assignment
    var origLoad = loaderInstance.load.bind(loaderInstance);
    loaderInstance.load = function(config) {
      return origLoad(Object.assign({}, config, {
        onEntrypointLoaded: async function(engineInitializer) { ... }
      }));
    };
    _loaderInstance = loaderInstance;
  }
});
```

`flutter_bootstrap.js` does `window._flutter.loader = new FlutterLoader` —
this triggers our setter, which wraps `load()` **before it is ever called**.
The guard `window._flutter.loader || (...)` then reads our getter (returns
the wrapped instance) and skips re-creating it.

### The Injection Sequence

```
initializeEngine()   → CanvasKit acquires WebGL context(s)    [Constraint 1 ✓]
                         (MapLibre not yet loaded)
inject maplibre-gl.js
await script.onload  → window.maplibregl is now defined        [Constraint 2 ✓]
runApp()             → first frame: MapLibreMap finds maplibregl
```

This satisfies all three constraints:
- Constraint 1 ✓ — CanvasKit has its context before MapLibre loads
- Constraint 2 ✓ — `maplibregl` is available before first Flutter frame
- Constraint 3 ✓ — setter intercepts the loader before `load()` is called

### Observability Signals (what to look for in `telemetry_logs`)

| Event | Meaning |
|-------|---------|
| `flutter_bootstrap_complete: maplibre_ready_before_runapp` | ✅ Correct path — setter worked, MapLibre loaded before runApp |
| `maplibre_lazy_loaded` | ✅ MapLibre script executed successfully |
| `app.cold_start` | ✅ Dart code running |
| `ios_fix: WEBGL_provoking_vertex applied` | ✅ MapLibre map initialised (iOS 18 WebKit bug workaround) |
| `maplibre_fallback_inject: flutter_bootstrap_never_fired` | ❌ Setter failed — something wrong with defineProperty approach |
| `maplibre_load_failed` | ❌ maplibre-gl.js failed to load (404, CORS, etc.) |
| `health: canvas element missing` | ❌ CanvasKit failed — WebGL context starvation or earlier crash |

## Remaining Known Issues

### `Script error. @ ?:0:0` (recurring)

Fires 2× per page load, ~2s after `visibility_visible`. Source is opaque
(`?:0:0` = cross-origin or service worker response without CORS headers).

**Candidates:**
- Flutter service worker serving stale `main.dart.js` from cache
- A third-party script loaded without `crossorigin` attribute
- The `maplibre-gl.js` script throwing before `maplibregl` is exported

**Not yet diagnosed.** Does not appear to block Flutter or MapLibre from loading.

### No Dart events in `telemetry_logs` for long-running sessions

Session `9a6cec99` was created April 3rd. JS beacon events appear but zero
Dart events appear post-April 3rd. Possible causes:

1. `ObservabilityService.flush()` silently failing (Supabase RLS or network)
2. `startPeriodicFlush()` timer not firing in backgrounded iOS tab
3. Session ID persists from localStorage but Supabase auth token has expired

`flush()` is fire-and-forget with `try/catch` that only `debugPrint`s on error.
Failures are invisible in `telemetry_logs`.

### `maplibre_version: not-loaded` always fires

The 500ms version-check timer fires before MapLibre is injected (injection
happens after `initializeEngine` which takes 2–4s). This is harmless — the
correct version appears in `maplibre_lazy_loaded`. The 500ms check can be
removed or pushed to after `flutter_bootstrap_complete`.

## Quick Reference: Boot Sequence (Correct)

```
HTML parse
  └─ inline <script> runs synchronously
       └─ defineProperty setter registered on window._flutter.loader

flutter_bootstrap.js loads (async)
  └─ window._flutter.loader = new FlutterLoader   → setter fires, wraps load()
  └─ _flutter.loader.load({ serviceWorkerSettings })
       └─ onEntrypointLoaded(engineInitializer)
            └─ initializeEngine()    → CanvasKit + WebGL context
            └─ inject maplibre-gl.js → await onload
            └─ runApp()              → first frame, MapLibreMap creates map
                 └─ app.cold_start (Dart)
                 └─ ios_fix: WEBGL_provoking_vertex applied
```
