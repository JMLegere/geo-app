# EarthNova v3 — Post-MVP Backlog

> Audited 2026-05-03. This is the prioritized post-MVP roadmap, not an
> implementation spec. Before coding any item, move the relevant details into the
> active design doc and acceptance criteria.

## Source-of-truth order

1. `AGENTS.md` — project rules, architecture constraints, and forbidden patterns
2. `docs/design.md` — MVP baseline and product direction
3. `docs/map-design.md` — current Map & Exploration source of truth
4. `docs/prd-game-systems.md` — aspirational game-system intent only
5. `docs/prd-game-systems-review.md` and `.agents/questions.md` — blockers that
   must be cleared before executing under-specified systems

## Roadmap recommendation

**Next area: Map & Exploration foundation.**

Reasoning:
- `docs/design.md` explicitly deferred Map, fog-of-war, and GPS as the next core
  gameplay items after MVP stability.
- `docs/map-design.md` now contains a dedicated source-of-truth map design.
- Discovery, daily seeds, sanctuary, recap, and many pack enhancements all depend
  on map position, cell detection, visit history, and fog state.
- Map foundation is less blocked than identification/enrichment, sanctuary,
  economy, NPC, or social systems.

Recommended next sub-scope:
- map screen and GPS-level map entry point
- GPS service, permission states, simulation/fallback behavior, and accuracy/ring state
- server-seeded Voronoi cell fetch/cache, render distance, and cell detail surface
- cell detection and `v3_cell_visits` recording
- fog-state computation from visit history; do not persist redundant fog snapshots
- map observability events and acceptance criteria

## Priority Band 1 — Recommended next: Map & Exploration foundation

Use `docs/map-design.md` as the active source of truth for these items; its
Strategy Cascade Roadmap breaks the work into initiatives, projects, and JTBD
user stories.

- [ ] Map screen — MapLibre GPS-level renderer, player marker, permission-required
      state, and tab-preserved map state
- [ ] GPS location service — real position stream, adaptive updates, simulation or
      fallback repository for tests/dev, and accuracy/ring-state thresholds
- [ ] Player marker/ring state — spline marker toward GPS position; suspend
      exploration while marker is in ring state
- [ ] Server-seeded Voronoi cells — fetch cells around player, cache static geometry,
      and support every point on Earth including ocean cells
- [ ] Cell detection — determine current cell from marker position and detect boundary
      crossings
- [ ] Visit recording — append every eligible visit to `v3_cell_visits`; first visit
      clears fog and can trigger encounter logic later
- [ ] Fog-state computation — derive Present/Explored/Nearby/render-distance states
      from GPS position, cell geometry, and visit history
- [ ] Cell detail surface — expose habitat labels in text so habitat is not color-only
- [ ] Map observability — log `map.cell_entered`, `map.cell_visited`,
      `map.fog_cleared`, `map.zoom_changed`, GPS degradation/restoration, data fetch,
      and render performance samples
- [ ] Acceptance criteria — add testable criteria before implementation continues

Deferred within Map:
- [ ] Zoom-out hierarchy polish — District/City/State/Country/World passport-style
      maps after the GPS-level foundation is stable
- [ ] Real tile-provider production decision — MBTiles/Mapbox/etc. after GPS-level
      behavior is proven
- [ ] Fog edge particles and advanced map juice — design polish after functional fog

## Priority Band 2 — Discovery & Encounters

Depends on Priority Band 1. Do not implement encounter progression until cell
entry, visit recording, and fog state are reliable.

- [ ] Daily seed system — `DailySeedService`, midnight GMT rotation, server RPC
      `ensure_daily_seed()`
- [ ] Deterministic encounters — `SHA-256(dailySeed + "_" + cellId)` selects species
      or loot for a visit
- [ ] Encounter count decision — resolve whether to adopt the backlog's old "3
      encounter slots per cell visit" or treat encounter rate as exploratory
- [ ] First-visit reveal — full species reveal/popup only after the encounter model is
      specified
- [ ] Revisit loot — toast/passive collection for repopulated cells after seed tiers
      are specified
- [ ] Seed freshness guard — pause or degrade discoveries after stale seed data
- [ ] Server validation — Edge Function or RPC validates encounter results and handles
      rollback/clawback
- [ ] Offline discovery — optimistic local discovery and queued validation only after
      online flow is stable
- [ ] Season/weather/time gates — defer until base encounter pacing is measured

## Priority Band 3 — Identification, Enrichment, and Species Cards

an execution plan, until the blockers below are resolved.

Execution blockers from `docs/prd-game-systems-review.md`:
- [ ] Stat model — decide species-truth vs species-baseline plus per-instance variance,
      including deterministic test vectors if variance is chosen

## Final recommendation

The next execution-ready area is **Map & Exploration foundation**.

Do not start Discovery, Identification, Sanctuary, Economy, NPC, or Social work as
if those systems are equally ready. They are downstream of the map foundation and/or
blocked by unresolved specs. A fresh execution session should begin by tightening
Map acceptance criteria against `docs/map-design.md`, then proceed through GPS,
cell detection, visit recording, and fog-state computation.
