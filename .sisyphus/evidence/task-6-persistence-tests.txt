================================================================================
ARCHITECTURE DECISIONS
================================================================================

1. LOCAL-FIRST PERSISTENCE
   - SQLite is the source of truth for all game state
   - Supabase sync is layered on top (Task 23)
   - Enables full offline functionality

2. REPOSITORY PATTERN
   - Clear separation between database and business logic
   - Each repository handles one domain (cells, collection, profile, sync)
   - Testable without database dependencies

3. SYNC QUEUE DESIGN
   - Separate table for pending changes
   - JSON-encoded data for flexibility
   - Supports insert/update/delete actions
   - Enables batch sync operations

4. SCHEMA MIRRORING
   - Local tables mirror Supabase structure
   - Simplifies sync logic (Task 23)
   - Consistent field names and types

5. UNIQUE CONSTRAINTS
   - (userId, cellId) unique on cell_progress
   - (userId, speciesId, cellId) unique on collected_species
   - Prevents duplicate records
